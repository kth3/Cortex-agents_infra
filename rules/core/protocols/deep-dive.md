# Deep-Dive Protocol (v1.0)

> [!IMPORTANT]
> 에이전트는 사용자가 모호한 버그를 제보하거나, 원인을 알 수 없는 시스템 오류가 발생했을 때 **즉각적인 코드 수정을 멈추고** 이 프로토콜을 활성화해야 합니다.

## 원칙: "탐색 우선, 구현 차중"
에이전트는 문제의 근본 원인을 증거로 입증하기 전까지는 단 한 줄의 기능 코드도 수정해서는 안 됩니다.

---

## Phase 1: Trace (인과관계 추적)
1. **가설 수립**: 발생한 현상의 원인에 대한 3가지 이상의 가능성을 제시합니다.
2. **코드 탐색**: `pc_capsule`, `pc_skeleton` 등을 사용하여 관련 모듈의 구조를 파악합니다.
3. **증거 수집**: 로그 분석, 재현 테스트 코드 작성 등을 통해 어떤 가설이 맞는지 검증합니다.
4. **결론 도출**: 문제의 지점(Hotspot)과 원인을 확정합니다.

## Phase 2: Interview (요구사항 구체화)
1. **현상 보고**: 파악된 원인을 사용자에게 간결하고 정확하게 설명합니다.
2. **해결책 제시**: 문제를 해결하기 위한 2~3가지의 대안(Trade-off 포함)을 제안합니다.
3. **의사결정 요청**: 사용자가 어떤 방향을 선호하는지, 추가적인 요구사항이 있는지 질문합니다.
4. **합의 도출**: 사용자의 승인을 받은 후, 해당 내용을 바탕으로 `pc_create_contract`를 수행하여 Phase 3(구현)로 넘어갑니다.

---

## 적용 트리거
- "이거 왜 안 돼?", "버그 좀 봐줘", "이상하게 돌아가" 등 모호한 문제 제기 시.
- 기존 코드를 파악하지 못한 상태에서 새로운 기능을 추가해야 할 때.
- 3회 이상 동일한 테스트 실패로 인해 원인 재분석이 필요할 때.
